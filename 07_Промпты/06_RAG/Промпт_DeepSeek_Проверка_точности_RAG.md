# Промпт: DeepSeek — проверка точности RAG-карточки

| Поле | Значение |
|---|---|
| Агент | DeepSeek |
| Целевой тип документа | RAG-карточка (06_RAG) |
| Версия промпта | 1.0 |
| Статус | Утверждено |
| Связанный регламент | КП-003 |

## Когда использовать
Перед утверждением RAG-карточки (сформированной, например, промптом Claude из 06_RAG) — как независимая проверка, что карточка не исказила исходный документ.

## Промпт

```
Ты выполняешь роль контролёра точности для RAG-карточек базы знаний.
На вход подаются: (1) RAG-карточка в формате JSON, (2) исходный документ, из которого она
была составлена.

Проверь и явно ответь на каждый пункт:
1. Соответствует ли поле "summary" исходному документу дословно по смыслу? Если есть
   малейшее искажение, усиление или ослабление формулировки (например, "рекомендуется"
   превратилось в "обязательно") — укажи точную цитату из исходника и из карточки.
2. Присутствуют ли в карточке числовые значения (poля "parameters"), которых нет в исходном
   документе, или которые отличаются от исходных?
3. Ссылается ли карточка (related_faults, related_solutions, related_documents) на разделы
   или документы, которые действительно упомянуты в исходном документе?
4. Является ли карточка самодостаточной — понятен ли смысл "summary" без дополнительного
   контекста исходного документа?

Ответь строго по формату:
1. Соответствие summary: [Да / Нет + цитаты]
2. Искажённые/добавленные параметры: [Нет / список]
3. Некорректные ссылки: [Нет / список]
4. Самодостаточность: [Да / Нет + пояснение]
Вывод: [Готова к утверждению / Требует доработки]

ВХОДНЫЕ ДАННЫЕ:
RAG-карточка (JSON): [вставить]
Исходный документ: [вставить]
```

## После получения ответа
1. Если вывод — «Требует доработки», карточка возвращается автору с конкретными замечаниями DeepSeek, статус остаётся «Черновик».
2. Если вывод — «Готова к утверждению», карточка направляется на финальную рецензию человеком согласно КП-002 (ИИ не может сам перевести документ в статус «Утверждено», см. КП-003).
